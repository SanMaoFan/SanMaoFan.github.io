浏览器解析资源的优先级

当浏览器开始解析网页，并开始下载图片、script 代码以及 css 样式等资源时，浏览器会为每个资源分配一个代表资源下载优先级的 fetch priority 标志。资源下载的顺序取决于这个优先级标志，这个标志的计算逻辑会受很多因素的影响：

  • script、css、font、image 等不同的资源类型会有不同的优先级。
  • 在 html 文档中引用资源的位置或顺序也会影响资源的优先级（例如在 viewport 中的图片资源可能具有高优先级，而在 link 标签中加载的、阻塞渲染的 css 则拥有更高的优先级）。
  • 有 preload 标志的资源有助于浏览器更快地发现资源并进行请求，也就是 preload 属性会提高资源请求优先级。
  • script 标签的 async、defer 属性会影响优先级。

在 chrome 浏览器中，可以通过 DevTools 中的 Network 选项卡下查看资源请求的优先级。

如果没有看到该列数据，可以在 Network 选项卡中点击右键，在 Header Options 项中选择要展示的数据列。

关于 preload

在 html 中使用 <link rel="preload" ... /> 来设定某些静态资源是否需要预先加载，以便它们在稍后渲染页面时立即可用。浏览器的预加载扫描器已经非常擅长此类做法，预加载通常最适合用于后来发现的资源———任何未由 html 直接加载的资源，例如通过内联样式属性加载的背景图像。但它对于浏览器可能没有按照你想要的优先顺序排列的任何其他内容也很有用。

例如：默认情况下， Chrome 会加载具有非常高优先级的字体，但如果某个用户的网络连接速度比较慢，它会使用后备字体并降低优先级。

script 脚本的优先级

但如果既希望脚本异步下载，又希望脚本拥有 “High” 的优先级又该如何处理呢？

一个可能的场景是将一个小型 SPA 嵌入到一个落地页的主要内容中。为了保持页面的 Core Web Vitals，特别是 LCP（最大内容绘制时间）和 FID（首次输入延迟），你需要将该脚本设置为高优先级（毕竟，它负责构建和驱动你的应用程序）。但与此同时，你不希望它阻碍页面的其余部分进行解析。

这个时候，fetchpriority 又起到了关键作用。

现在，它以更高的优先级下载，同时仍然不会阻止页面的其余部分：

优先级越高，异步脚本加载的越快，在这种情况下，异步脚本甚至可能在同步和内联脚本之前加载。

同样的思路，fetchpriority 也可以用来延迟脚本加载。

当项目中预先知道脚本的优先级，并且浏览器可能也没有足够的信息来自行确定时，可以将 fetch priority 添加到脚本上。它对于还想以非阻塞、异步方式加载的脚本的优先级特别有帮助

最后

当然，fetchpriority 也不能过度使用，因为加多了等于什么都没加，甚至会导致网页性能下降。有明确知道哪些资源的优先级，对于网页的加载体验还是有很大帮助的。

但当前 fetchpriority 还是实验性产物，期待后续能有更好的发展，具体可前往 <script>：脚本元素 链接查看浏览器的兼容情况。
