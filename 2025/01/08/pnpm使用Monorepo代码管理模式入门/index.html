pnpm使用Monorepo代码管理模式入门

发表于 2025-01-08 | 分类于 前端

介绍

在传统开发中，如果一个项目组中要同时开发后台管理系统和前台展示系统，按照以往思路是使用两个代码仓库进行维护，分别存放后台管理系统和前台展示系统，但有时候可能同一个版本依赖或者公用组件、方法，在两个系统同时用到了，但却需要复制一份到另一个项目，如果其中一个项目公用方法改动，另一个项目也得跟着改，这从而造成了代码维护性差，维护困难的问题。

而这里介绍的 Monorepo 可以解决以上问题，还能带来其他便利，首先， Monorepo 是一种项目代码管理方式，指的是单个仓库管理多个项目，有助于简化代码共享、版本控制、构建和部署等方面的复杂性，并提供更好的可重用性和协作性。

关于 Monorepo 的演进

阶段一：单仓库巨石应用。 在一个 Git 仓库维护着项目代码，所有功能模块都集中在一个单一的应用中，它的优点在于简单易懂、部署方便，适用于初期小规模项目。随着迭代业务复杂度的提升，项目代码会变得越来越多，越来越复杂，其代码构建的效率也会降低，最后导致成了一个单体巨石引用，这种代码管理方式称为 Monolith；

阶段二：多仓库多模块应用。 再后来，将项目拆解成了多个业务模块，并在多个 Git 仓库管理，,模块实现了解耦，降低了巨石应用的复杂度，每个模块可以独立编码、测试、发版，这样实现了每个仓库独立管理代码和依赖，代码维护难度降低了，构建效率也得以提高，每个项目可以选择适合的工具和依赖版本，也容易对项目进行精细化的权限管理，但其在不同项目之间共享代码和资源较为困难，可能导致重复代码，跨项目的依赖管理复杂，需要额外的工具和配置，团队之间可能需要在多个仓库之间切换，可能降低工作效率，在介绍中的举的例子就像这种做法，这种代码管理方式称为 MultiRepo；

阶段三：单仓库多模块应用。 在考虑到 MultiRepo 管理方式中跨仓库代码难以共享，分散在单仓库的模块依赖管理复杂（底层模块升级，会牵动上层依赖变动，更新不及时可能造成开发阻塞）；为了方便多个项目使用共同资源，于是将多个项目集成到一个仓库下，共享工程配置，同时又快捷的共享模块代码，在这种模式下，所有项目共享版本控制历史，促进代码共享和集中管理，也简化了依赖管理，项目成员在同一个仓库中协作，代码审查、问题跟踪和变更管理也变得更加方便，这种代码管理方式称为 Monorepo；

在以往的开发方式中，安装依赖包一般使用 npm、yarn、cnpm，但还有 pnpm，其为一个老牌的 node.js 包管理器，支持 npm 的所有功能，完全足以替代 npm。它采用全局存储，每个项目内部使用了硬连接，会比以往的包管理器省空间，安装速度也快。而最重要的是，pnpm 是支持 Monorepo 的，也就是说，全局安装了 pnpm 包管理器，也就可以直接在项目使用 Monorepo 代码管理方式了。

Monorepo 能解决的问题

幽灵依赖

在使用 npm/yarn 安装依赖时，可能存在依赖提升，例如某个项目使用的依赖，明明没有在其 package.json 文件中声明，但却可以直接使用，这种现象便是“幽灵依赖”。

举个例子：前端开发项目中引用了 A 库，A 库引用了 B 库，B 库引用了 C 库，由于 npm 的特性会将 B 和 C 进行扁平化处理，导致我们在项目制可以不安装 B 和 C 的情况下直接使用 B 和 C 库，也就是在项目中可以直接 import bFn from 'B' 是允许的，但是在后续当 A 库更新以后不使用 B 库，那么我们开发的项目就会出现 B，C 库引用报错问题。

而 pnpm 采用了非扁平目录，在项目中是不能使用到其依赖包所依赖的包，避免了幽灵依赖问题。

依赖安装时间过长

在以往项目中，如果项目所需要的依赖过多，那么重新给项目 install 时，其时间可能耗时，况且当多个项目都安装依赖，其等待会更痛苦。而在 Monorepo 中，虽然每个项目都有自己的 package.json 依赖列表，但是在 pnpm 操作下，可以将相同版本的依赖提升到 Monorepo 的根目录下，减少冗余依赖的安装。并且 pnpm 实现全局缓存，硬连接包可以实现空间节省、安装快速

构建打包耗时长

多个项目在构建任务存在依赖时，往往是串行构建或者全量构建，导致构建时间较长；而 Monorepo 采用增量构建，也可以将串行构建优化成并行构建。
